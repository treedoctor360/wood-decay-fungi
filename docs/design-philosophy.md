# 設計思想（このシステムをなぜこう作るか）

樹木医向け「木材腐朽菌 同定アシスト」の**設計の背骨**をまとめる。
「何をどう直すか」の運用ルールは `CLAUDE.md`、時点ごとの変更は `docs/session-summary-*.md` を参照。
ここは**判断の拠り所となる思想（なぜ）**を書く場所であり、機能追加のたびにブレさせない。

---

## 1. 人間が判定者、AIは助手（Human-in-the-loop）
最上位の原則。**AIは「候補出し」と「観察ガイド」の助手であり、判定者ではない。最終判定は樹木医（人）が行う。**
- だからAIの出力は「確定」ではなく候補＋確信度＋理由＋「未確認の決め手」の形を取る。
- 迷う場合は断定せず `hantei` を「保留」にし、確信度は控えめにする（`App.jsx` プロンプト方針8-9）。
- 誤同定そのものより、**根拠の弱い断定**を避けることを優先する。UIにも「最終判定は樹木医が検索表で確定」と明記する。

## 2. 書籍が唯一の根拠（推測で埋めない）
データの正しさは**『緑化樹木腐朽病害ハンドブック』(日本緑化センター, 2017)**に忠実であることで担保する。
- `speciesMaster`（全65種）は書籍に忠実。**「記載なし」は書籍がその項目を書いていない場合＝空欄が正**。それらしい値を推測で補完しない。
- 種データを直すときは、必ず**書籍本文の該当種と和名で照合**してから（OCRの崩れがあるため実物ページ優先）。
- 出典を `shutten` で明示し、根拠をたどれる状態を保つ。

## 3. 専門家の同定ロジックを骨組みにする（検索表ファースト）
属性の類似度マッチングではなく、**図鑑編者の同定ロジック（簡易検索表 p.9-11）を構造化して判断の骨組みにする**。
- `src/data/searchKey.js` が検索表を群（A〜D、下位D1〜D3）として構造化。観察状況（宿主・部位）に**該当する群だけ**をプロンプトに原典どおり注入する（`keyPromptBlock`）。
- プロンプトは「この分岐に沿って絞る。分岐条件と矛盾する種は色やサイズが似ていても外す／確信度を極小化する」と指示（`searchKey.js` L212-215、`App.jsx` L830-846）。
- **意図的な設計**: 見た目の印象より検索表の階層を優先させることで、素人的な「色形が似ている」誤りを防ぐ。

## 4. 階層的に絞り込む（上位の矛盾は見た目に勝つ）
判定は次の優先順位で上位から絞る（`App.jsx` L833-845）。
```
(1) 宿主樹種（針葉樹/広葉樹）
(2) 発生部位（根株 / 幹・枝）
(3) 寿命・質感（一年生・軟質 / 多年生・硬質）
(4) 裏面の形態（ヒダ / 管孔 / 針・歯牙 / 平滑）
(5) 組織色・腐朽型（濃色→サルノコシカケ科系, 黄色系→タバコウロコタケ科, 白色/褐色腐朽）
```
- **上位階層で矛盾する種は、写真がどれほど似ていても候補に挙げない。**
- ただし各段が「不明・入力なし」のときは、その段は判断に使わない（減点しない）。空欄を根拠に外さない、という非対称ルール。

## 5. 「写真で見える」と「見えない決め手」を分ける
腐朽菌の確定には、写真に写らない情報（断面の肉色・KOH反応・剛毛体・匂い・触感・宿主特異性・顕微鏡下の胞子/菌糸）が要ることが多い。設計はこれを正面から扱う。
- 各種データを `mieru`（肉眼で見える特徴）と `kettede`（決め手）に分け、AIには候補ごとに `mieru`（写真から見えた）と `miketsu`（未確認の決め手）を分けさせる。
- `micro:true` の種や、KOH/メルツァー/担子胞子など現地で見えない項目を含む種は、理由に「※現地観察のみでは困難・採取して顕微鏡確認を推奨」と明記させる。
- 決め手が拮抗するときは `kagi`（見分けの鍵：拮抗2種・決定的な違い・今すぐ確認できる方法）を必ず埋め、**現地で確認できる方法を顕微鏡より優先**して提示する。
- **設計上の帰結（重要）**: 決め手が断面色・宿主特異性・顕微鏡にしかない種は、表面写真だけでは原理的に上位に出せないことがある。この場合は誤断定せず「保留＋確認質問」に落とすのが正しい振る舞い。精度は「断面を削る／宿主を答える」といった**現地確認との往復**で上げる設計であって、写真一枚で確定させる設計ではない。

## 6. 候補は65種のクローズドセット（越境しない）
- AIは `speciesMaster` 全65種の中からのみ候補を選び、**リスト外の菌は絶対に挙げない**（テングタケ・シイタケ等の一般キノコは未収録）。ハルシネーションによる架空種を防ぐ。
- 一方で現実にはリスト外の種もある。そのため**手動登録**（65種から選ぶ／自由入力）で樹木医が確定でき、`jouhougen`（`from_ai`/`from_master`/`from_manual`）で情報源を残す。閉じたAI候補と開いた現実を、記録の透明性で橋渡しする。

## 7. 静的サイト + GAS中継（鍵をクライアントに置かない）
- React + Vite の**静的サイト**を GitHub Pages で公開。サーバーを持たない。
- Gemini API キーは**GAS（Apps Script）中継**に隠し、クライアントJSには置かない。GASがキーを保持して中継する。
- 参照元制限は原理的に不可（GASはOrigin/Refererを読めない）。**主防御はGAS側の1日レート制限＋無料枠キー＋任意トークン＋Cloud予算上限**という多層の“薄い壁”で、本気の攻撃者ではなくボット・誤用・暴発コストを抑える現実解。
- GASはリポジトリからデプロイされない（`gas/*.gs` は参照コピー）。これは「秘密をGit管理外に置く」原則の帰結。

## 8. 端末内保存を第一に、共有はゆるく（Local-first + 合言葉共有）
- 記録はまず端末内 IndexedDB に永続保存（オフラインでも記録できる）。その上で GAS DB と同期する。
- 全ユーザーが同じスプレッドシートを共有し、**共有キーワード（`group`）で仕分け**る。同じ合言葉の人だけで記録を共有・削除できる。
- キーワードは“グループの合言葉”であり**個人認証ではない**。厳密な本人確認より、樹木医コミュニティでの手軽な共有を優先した割り切り。

## 9. 透明性と振り返りを記録に残す
- AIの所見 `shoken`（写真から見えた特徴の自然文）、候補ごとの `riyu`（理由）、情報源 `jouhougen` を記録に保存する。
- 目的は**後からの精度分析**。AIが外した／候補に無かった事例を樹木医の確定と突き合わせ、データや検索表・プロンプトを改善する材料にする。判定を隠さず、間違いを追跡可能にすることが精度向上の前提。

---

## この思想が生む既知のトレードオフ
- **写真だけで到達しにくい種がある**（例: 決め手が断面の肉色・宿主特異性・デキストリノイド反応にしかない種）。これは原則5・4の帰結であり、保留と確認質問で受けるのが正しい。精度を上げたいときは「見た目の一致」を足すのではなく、**現地で確認できる決め手をどう引き出すか**（宿主特異性の加点、断面確認の誘導、確認質問の設計）で改善する。
- **クローズドセット（原則6）**ゆえにリスト外の種は原理的に出ない。手動登録と記録の情報源タグで補う。
- **ゆるい共有（原則8）**ゆえに厳密なアクセス制御はない。必要になればログイン導入は別スコープ。
