# GAS（Google Apps Script）構成

このアプリのサーバー側は **役割ごとに分離した2つの独立プロジェクト** で構成する。
1プロジェクト＝1責務＝`doGet`1個/`doPost`1個。ここに混在させないことで、
以前のような「1つのApps Scriptにコードが乱立して整理不能」を防ぐ。

| ファイル | プロジェクト | 役割 | フロント側の定数 |
|---|---|---|---|
| `gemini-relay.gs` | Gemini中継 | 画像＋プロンプトをGeminiに中継 | `GAS_URL` |
| `records-db.gs` | 記録DB中継 | 記録をスプレッドシートへ保存/取得 | `GAS_DB_URL` / `DB_TOKEN` |

## 作り直し手順

各 `.gs` は **別々の新規Apps Scriptプロジェクト** に貼る。

### 1. Gemini中継（gemini-relay.gs）
1. 新規Apps Scriptプロジェクトを作り、`gemini-relay.gs` の内容を貼る。
2. プロジェクトの設定 → スクリプト プロパティ:
   - `GEMINI_API_KEY` … Geminiのキー本体（**必須**）。無料枠キー推奨。
   - `SHARED_TOKEN` … 任意。設定するとトークン照合が有効。フロントの
     `RELAY_TOKEN` に同じ値を入れる。
   - `DAILY_LIMIT` … 任意。1日の中継上限（既定300）。
3. デプロイ → 新しいデプロイ → 種類「ウェブアプリ」、アクセスできるユーザー
   「全員」。発行された `/exec` URL をフロントの `GAS_URL` に貼る。

### 2. 記録DB中継（records-db.gs）
1. 記録用スプレッドシートを用意（1枚目のシート名 `records`、1行目に
   ヘッダ列。`record_id` 列を必ず含める。列はアプリのフィールド名に対応）。
2. そのスプレッドシートに紐づくApps Script（拡張機能 → Apps Script）か、
   独立プロジェクト＋スクリプトプロパティ `SHEET_ID` で対象を指定。
3. `records-db.gs` の内容を貼る。
4. スクリプト プロパティ:
   - `TOKEN` … **必須**。フロントの `DB_TOKEN` と一致させる共有トークン。
   - `SHEET_ID` … 独立プロジェクトの場合に対象スプレッドシートID。
   - `SHEET_NAME` … 任意（既定 `records`）。
5. デプロイ → ウェブアプリ → 発行URLをフロントの `GAS_DB_URL` に貼る。

## 露出URL対策の考え方（重要）

静的サイト（GitHub Pages）では、ブラウザが叩くURLは必ず露出する。よって
「URLを隠す」ではなく「露出しても被害が出ない/限定される」設計にする。

- **Gemini中継**: ①GAS側1日レート制限（本命） ②任意トークン（ボット除け）
  ③無料枠キー＋Cloud予算/割当上限で金銭被害を原理的にゼロにする。
  ※GASのWeb AppはOrigin/Refererヘッダを読めないため、参照元制限は不可。
- **記録DB**: GETにもトークンを必須化（旧版はGET素通しで全記録が読めた）。
  トークンはフロントJSに見えるので、機微情報は保存しない前提で運用する。
